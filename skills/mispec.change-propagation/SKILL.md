# Spec 变更传播

在任意阶段修改需求，自动同步到所有受影响的制品。

## 触发场景

- 用户在 specify/clarify/plan/tasks/implement **任意阶段**说"我想修改..."、"需求变了"、"调整一下"、"这里不对"
- 用户在实施阶段发现规范与实际不符，需要回溯修改
- 用户在查看 plan 后说"这里的设计我想改"
- 用户在看 tasks 时说"这个任务的范围不对"

---

## 执行步骤

### 步骤 1: 加载当前制品状态

从仓库根目录运行 `.specify/scripts/powershell/check-prerequisites.ps1 -Json -IncludeTasks` 并解析 JSON，获取：

- `FEATURE_DIR`: 功能目录绝对路径
- `ARCHIVE_DIR`: 归档目录绝对路径
- `AVAILABLE_DOCS`: 已存在的制品列表

扫描已存在的制品文件（仅读取已存在的，不假设所有都存在）：

| 制品 | 路径 | 制品层级 |
|------|------|---------|
| 功能规范 | `FEATURE_DIR/spec.md` | L1（最上游） |
| 实施计划 | `FEATURE_DIR/plan.md` | L2 |
| 数据模型 | `FEATURE_DIR/data-model.md` | L2.5（可选） |
| 任务列表 | `FEATURE_DIR/tasks.md` | L3 |
| 研究归档 | `ARCHIVE_DIR/research.md` | L2（辅助） |
| API 合同 | `ARCHIVE_DIR/contracts/` | L2.5（辅助） |

记录哪些制品已存在，构建**变更传播图**（仅针对已存在的制品）。

---

### 步骤 2: 理解修改意图

理解用户描述的修改内容，从以下维度分析：

#### 2a. 识别修改的源头层级

根据用户描述，判断修改发生在哪一层：

| 用户表述 | 修改层级 | 示例 |
|---------|---------|------|
| "需求/功能/用户故事要改" | L1 spec | "用户不需要注册，改为直接用手机号登录" |
| "技术方案/架构要改" | L2 plan | "不用 Redis，改用内存缓存" |
| "数据结构/字段要改" | L2.5 data-model | "用户表加一个 avatar 字段" |
| "任务/实现细节要改" | L3 tasks | "这个任务拆成两个" |
| "实施中发现问题" | 实施反馈 | "按 plan 做不通，需要调整" |

#### 2b. 提取修改的具体内容

分析修改涉及：

- **变更类型**: 新增 / 删除 / 修改 / 范围调整
- **影响实体**: 哪些功能、数据实体、API、任务受到影响
- **影响范围**: 局部（单个功能点）/ 结构性（跨多个模块）/ 根本性（改变核心逻辑）

#### 2c. 确认变更意图（必要时向用户确认）

若修改意图不明确（如"这里改一下"但未说明改什么），向用户提问：

> **请确认修改内容：**
>
> 你想修改的是：
> - A. [从上下文推断的选项一]
> - B. [从上下文推断的选项二]
> - C. 描述你具体想改什么

收到确认后继续。若意图清晰，直接进入步骤 3。

---

### 步骤 3: 构建变更影响分析

基于修改层级和已存在的制品，构建**影响分析矩阵**：

```
变更源: [L1 spec / L2 plan / L2.5 data-model / L3 tasks / 实施反馈]
修改内容: [简要描述]

受影响制品:
┌─────────────────┬──────────┬─────────────────────────────┐
│ 制品             │ 影响程度  │ 需要同步的具体内容           │
├─────────────────┼──────────┼─────────────────────────────┤
│ spec.md         │ 直接/间接 │ [具体章节和内容]             │
│ plan.md         │ 直接/间接 │ [具体章节和内容]             │
│ data-model.md   │ 直接/无   │ [具体字段或实体]             │
│ tasks.md        │ 直接/级联 │ [受影响的任务 ID 列表]       │
│ research.md     │ 参考/无   │ [需要更新的决策记录]         │
│ contracts/      │ 直接/无   │ [受影响的 API 端点]          │
└─────────────────┴──────────┴─────────────────────────────┘
```

**级联规则**：

- L1 spec 修改 → 影响所有下游制品（plan、data-model、tasks）
- L2 plan 修改 → 影响 data-model、tasks，**不自动修改 spec**（除非修改揭示了 spec 问题）
- L2.5 data-model 修改 → 影响 tasks 中相关数据操作任务
- L3 tasks 修改 → 仅修改 tasks.md，不影响上游
- 实施反馈 → 根据问题根因决定回溯层级，可能影响 plan 或 spec

**特殊情况**：若 L2/L3 的修改揭示了 spec 中的根本性问题（如发现需求本身有矛盾），需标记为 **"上游影响"** 并在步骤 4 中优先处理。

向用户展示影响分析，等待确认：

> **变更影响分析：**
>
> [展示上方矩阵]
>
> 将按以下顺序更新制品：[制品列表，从上游到下游]
>
> 是否继续？(回复"是"或"继续"，或指定跳过某个制品)

---

### 步骤 4: 执行变更传播

按**从上游到下游**的顺序依次更新受影响的制品。每个制品更新完成后继续下一个，不等待用户确认（除非遇到冲突）。

#### 4a. 更新 spec.md（如果受影响）

读取当前 `spec.md`，定位需要修改的章节：

- 修改功能需求、用户故事、成功标准、假设、边缘情况等
- **保留不受影响的章节不变**
- 在文件末尾的 `## 变更记录` 章节（如不存在则创建）追加本次变更：

```markdown
## 变更记录

### [日期] - 变更摘要

**变更来源**: [用户在 specify/plan/tasks/implement 阶段提出]
**变更类型**: [新增/删除/修改/范围调整]
**变更内容**: [简要描述]
**受影响章节**: [章节列表]
```

写回 `spec.md`。

#### 4b. 更新 plan.md（如果受影响）

读取当前 `plan.md`，根据 spec 的变化更新：

- 技术方案、架构选择、数据模型引用、阶段划分等
- 若技术方案需要重大调整，标记 `[NEEDS REVIEW: <具体问题>]` 并在报告中列出
- **保留不受影响的技术决策**
- 在 plan.md 末尾追加变更记录

写回 `plan.md`。

#### 4c. 更新 data-model.md（如果受影响且文件存在）

读取当前 `data-model.md`，根据变更更新：

- 新增/删除/修改实体、字段、关系、状态机
- 保持字段描述和验证规则的一致性
- 标注新增字段和删除字段（删除字段用 `~~字段名~~` 标注，待任务完成后清理）

写回 `data-model.md`。

#### 4d. 更新 tasks.md（如果受影响且文件存在）

这是最复杂的更新，需要精确处理依赖关系。

**任务影响分类**：

| 影响类型 | 处理方式 |
|---------|---------|
| 直接受影响（任务内容需修改） | 更新任务描述、inputs/outputs |
| 依赖受影响（依赖的任务改变了） | 更新 `#meta.deps` |
| 需要新增任务 | 在适当阶段插入新任务，重新编号 |
| 需要删除任务 | 标记为 `~~T00X~~ [已废弃: 原因]`，不立即删除 |
| 已完成任务受影响 | 标记警告 `[COMPLETED BUT CHANGED]`，提示可能需要回溯 |

**重要规则**：

- **已完成任务（`[x]`）受影响时**：不修改已完成状态，在任务后追加注释 `<!-- CHANGED: [变更内容] -->` 并在报告中列出，让用户决定是否需要重做
- **维护 Execution DAG 一致性**：任何 `#meta.deps` 变更后，必须同步更新 `## Execution DAG` 章节
- **任务编号连续性**：新增任务尽量插入合适位置，若无法保持连续性则在末尾追加并在 DAG 中反映正确依赖

写回 `tasks.md`。

#### 4e. 更新 archive/ 辅助制品（如果受影响）

- `research.md`: 在末尾追加决策更新记录，不修改已有决策（除非明确废弃）
- `contracts/`: 更新受影响的 API 端点定义

---

### 步骤 5: 一致性验证

所有制品更新完成后，执行快速一致性校验：

1. **术语一致性**: 检查各制品中同一概念是否使用相同术语
2. **引用完整性**: tasks.md 引用的文件路径是否仍然有效
3. **依赖完整性**: tasks.md 的 `#meta.deps` 中引用的 TaskID 是否都存在
4. **DAG 一致性**: Execution DAG 边是否与 `#meta.deps` 严格一致

若发现不一致，立即修复后继续。

---

### 步骤 6: 输出变更摘要报告

向用户展示：

```markdown
## 变更传播完成

**变更摘要**: [一句话描述]
**变更源**: [哪个阶段触发，修改了什么]

### 已更新的制品

| 制品 | 变更内容 | 状态 |
|------|---------|------|
| spec.md | [具体变更] | ✅ 已更新 |
| plan.md | [具体变更] | ✅ 已更新 |
| data-model.md | [具体变更] | ✅ 已更新 |
| tasks.md | [受影响任务列表] | ✅ 已更新 |

### 需要关注的事项

#### 已完成任务受到影响（需人工决策）
[列出 `[COMPLETED BUT CHANGED]` 的任务，说明变更内容和建议操作]

#### 需要人工审查的决策
[列出标记了 `[NEEDS REVIEW]` 的内容]

#### 跳过的制品
[列出因不受影响而跳过的制品]

### 建议的下一步
[根据当前阶段给出建议：继续实施 / 重新运行 analyze / 等]
```

---

## 行为规则

1. **最小影响原则**: 只修改真正受影响的内容，不受影响的章节一律保留原样
2. **上游优先**: 始终从最上游制品开始更新，确保下游基于最新上游生成
3. **已完成不回滚**: 已完成的任务不自动撤销，但会标记警告让用户决定
4. **追加变更记录**: 每次变更都在相关制品末尾记录，保持可追溯性
5. **冲突时暂停**: 若某制品的更新会产生矛盾（如两个需求互相冲突），暂停并询问用户如何解决
6. **实施阶段回溯**: 若在 implement 阶段发现问题，明确告知需要回溯到哪一层，以及已完成的任务是否需要重做
7. **仅操作已存在的制品**: 不为用户自动创建尚未生成的制品（如 plan 还没有时，不自动创建 plan.md）

---

## 实施阶段的特殊处理

当变更来源是"实施阶段发现问题"时，需要额外的根因分析：

**判断回溯层级**：

```
实施问题描述: [用户描述]

根因判断:
- 若是"按 plan 做不通，技术方案有误" → 回溯到 L2 plan
- 若是"spec 需求本身有矛盾/遗漏" → 回溯到 L1 spec
- 若是"数据字段设计有问题" → 回溯到 L2.5 data-model
- 若是"任务拆分不合理" → 只修改 L3 tasks
```

**向用户说明影响**：

> **实施阶段变更检测**
>
> 根据你描述的问题，根因在 [层级]，需要修改 [制品列表]。
>
> 已完成的相关任务：[列表]
> 这些任务可能需要重做。
>
> 是否继续传播变更？

---

## 不触发此 Skill 的场景

- 正常的 mispec 流程推进（specify → plan → tasks → implement）
- 纯粹的代码实现问题（不涉及需求或设计变更）
- 在 specify 阶段的正常澄清（使用 /mispec.clarify）
